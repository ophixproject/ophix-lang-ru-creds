На этой странице описан переход с пакета `cred-server` до реструктуризации (метка приложения `creds`) на текущий пакет `ophix-creds`. Если вы устанавливали ophix-creds с нуля, эта страница не применима.

---

## Что изменилось

Старый `cred-server` и текущий `ophix-creds` структурно несовместимы — запустить `migrate` на старой базе данных не получится. Ключевые отличия:

| Аспект | Старый (`cred-server`) | Новый (`ophix-creds`) |
| --- | --- | --- |
| Метка приложения | `creds` | `ophix_creds` |
| Имена таблиц | `creds_*` | `ophix_creds_*` / `ophix_core_*` |
| Модели Host / Client | В приложении `creds` | В `ophix_core` (ophix-server-base) |
| Хранение `secret_json` | Открытый JSONField | Зашифрованный TextField (Fernet) |
| `ClientCredential` | Только `enabled`, `notes` | + `can_update`, `can_delete`, `can_share` |

Столбцы `can_update`, `can_delete` и `can_share` импортируются со значением `False` — права доступа при миграции не повышаются.

---

## Порядок перехода

Рекомендуемый подход — **параллельная миграция**: оставить старый сервер работающим до полной проверки нового, затем вывести его из эксплуатации.

### 1. Развернуть новый credserver

Следуйте стандартному руководству по установке — новое виртуальное окружение, новая база данных:

```bash
pip install ophix-server-base ophix-creds
ophix-manage configure_install credserver
ophix-manage run_install credserver
sudo bash credserver_sudo_install.sh
```

### 2. Проверить работу нового сервера

Откройте административную панель по адресу `https://ваш.новый.сервер/admin/` и подтвердите применение миграций:

```bash
ophix-manage migrate --check
```

### 3. Выполнить команду импорта

```bash
ophix-manage import_legacy_credserver \
    --db-host <старый-db-host> \
    --db-name <старое-имя-бд> \
    --db-user <пользователь-бд> \
    --db-password <пароль-бд>
```

Сначала выполните `--dry-run`:

```bash
ophix-manage import_legacy_credserver --db-name credserver_db --dry-run
```

Для источника PostgreSQL:

```bash
ophix-manage import_legacy_credserver --db-engine postgres --db-name ...
```

Команду можно безопасно запускать повторно — существующие записи пропускаются с предупреждением.

### 4. Проверить импортированные данные

Войдите в новую административную панель и убедитесь, что:

- Все хосты отображаются в разделе **Клиенты и хосты → Хосты**
- Все клиенты отображаются в разделе **Клиенты и хосты → Клиенты**
- Все учётные данные отображаются в разделе **Учётные данные**
- Связи клиент–учётные данные сохранены

### 5. Обновить конфигурацию клиентов

```bash
cred-client set server https://новый.сервер.hostname
cred-client download ca-cert   # если CA TLS изменился
```

### 6. Вывести старый сервер из эксплуатации

После того как все клиенты проверены на новом сервере, остановите и удалите старый.

---

## Примечания

- **Токены сохраняются.** Значения `api_token` переносятся без изменений — клиенты подключаются без повторной регистрации.
- **Шифрование.** `CRED_ENCRYPTION_KEY` на новом сервере не связан со старым. Все импортированные учётные данные перешифровываются новым ключом. Храните этот ключ в надёжном месте.
- **Новые столбцы прав.** `can_update`, `can_delete` и `can_share` при импорте равны `False`. Проверьте их в панели администратора, если требуется клиентская запись.
- **Старая база данных никогда не изменяется** в процессе импорта.
