Домен учётных данных хранит именованные JSON-секреты и распространяет их авторизованным клиентам по HTTPS. Секреты извлекаются по запросу и используются исключительно в памяти — на диск клиента они никогда не записываются.

Каждая запись учётных данных — это именованный объект, содержащий произвольный JSON (`secret_json`). Доступ контролируется на уровне клиента: клиент может получить только те учётные данные, к которым он был явно привязан администратором.

---

## Шифрование хранимых данных

Поле `secret_json` хранится в базе данных в зашифрованном виде с использованием симметричного шифрования Fernet (AES-128-CBC с HMAC-SHA256). В столбце базы данных хранится зашифрованный токен — открытый секрет никогда не записывается на диск в читаемом виде.

Шифрование и расшифровка происходят прозрачно: административная панель, API и клиентские инструменты работают с открытым значением JSON.

### Первоначальная настройка

Перед первым запуском `migrate` сгенерируйте ключ шифрования:

```bash
ophix-manage generate_cred_key
```

Команда выводит ключ. Добавьте его в `.env`:

```ini
CRED_ENCRYPTION_KEY=<сгенерированный ключ>
```

Затем выполните миграции:

```bash
ophix-manage migrate
```

Миграция шифрует все существующие записи в открытом виде. Если `CRED_ENCRYPTION_KEY` не задан, миграция остановится с понятным сообщением об ошибке до внесения каких-либо изменений.

### Управление ключами

- Ключ — это 32-байтный ключ Fernet, хранящийся в виде URL-safe base64 в `.env`
- **Создавайте резервную копию ключа отдельно от базы данных.** Потеря ключа означает потерю доступа ко всем сохранённым учётным данным

### Ротация ключа

Для замены ключа шифрования и повторного шифрования всех учётных данных:

```bash
ophix-manage rotate_cred_key
```

Используйте `--no-input` для плановой автоматической ротации.

## cred-client

`cred-client` — клиент командной строки первого уровня для сервера учётных данных Ophix. Конфигурация хранится в `.cred.env`.

| Переменная | Описание |
| --- | --- |
| `CREDSERVER_URL` | Базовый URL сервера учётных данных |
| `CREDSERVER_CA_CERT` | Путь к CA-сертификату сервера |
| `CREDSERVER_API_TOKEN` | 64-символьный шестнадцатеричный API-токен |

### Установка

```bash
pip install ophix-cred-client
```

### Инициализация

```bash
# В один шаг
cred-client quickstart https://credserver.internal мой-клиент

# Пошагово
cred-client set server https://credserver.internal
cred-client download ca-cert
cred-client register мой-клиент
```

### Ротация токена

```bash
cred-client rotate-token
```

Проверяет новый токен перед перезаписью `.cred.env`. Можно запускать из cron.

### Получение учётных данных

```bash
cred-client fetch db_prod         # выводит JSON в stdout
```

### Проверка

```bash
cred-client check --all                    # проверить все сопоставленные учётные данные
cred-client check --all --verbose          # показать детали ошибок
cred-client check --var DB_PROD_CRED_NAME  # проверить по имени переменной окружения
cred-client check --name db_prod           # проверить по имени учётных данных
```

## Управление учётными данными в панели администратора

Учётные данные и доступ клиентов управляются через административную панель Django:

- **Учётные данные** — создание и редактирование учётных данных, просмотр клиентов с доступом
- Страница сведений о **Клиенте** — управление доступными клиенту учётными данными через встроенную форму

При привязке клиента к учётным данным флажок **enabled** на записи связи управляет активностью доступа. Отключённые связи отображаются в списке администратора курсивным приглушённым текстом.

## Справочник API

Все запросы требуют `Authorization: Token <api_token>` и должны поступать с зарегистрированного IP-адреса хоста.

### Получить учётные данные

```http
GET /api/credentials/<name>/
```

### Создать учётные данные

```http
POST /api/credentials/<name>/
Content-Type: application/json

{
  "secret_json": { "KEY": "значение" },
  "description": "Необязательное описание"
}
```

### Обновить учётные данные

```http
PUT /api/credentials/<name>/
```

Требует `can_update` на записи связи.

### Удалить учётные данные

```http
DELETE /api/credentials/<name>/
```

Требует `can_delete` на записи связи **и** `ENABLE_ARTIFACT_DELETE=true` в `.env`.

## Настройки сервера

| Переменная | По умолчанию | Описание |
| --- | --- | --- |
| `CRED_ENCRYPTION_KEY` | _(обязательно)_ | Ключ шифрования Fernet. Генерируется командой `ophix-manage generate_cred_key`. |
| `ENABLE_ARTIFACT_DELETE` | `False` | Разрешить клиентам удалять собственные учётные данные. |
| `AUTH_LEAK_INFO` | `False` | Включать детали ошибок в ответы API. Только `True` для разработки. |
| `MINIMUM_TOKEN_ROTATE_TIME` | `3600` | Минимальный интервал между ротациями токена в секундах. |

## Контроль доступа

Каждый запрос проходит четырёхуровневую проверку:

1. `Host.enabled` — хост-машина зарегистрирована и активна
2. `Client.enabled` — конкретный клиентский процесс активен
3. `Credential.enabled` — сами учётные данные активны
4. `ClientCredential.enabled` — данный клиент получил доступ к этим учётным данным

Если хотя бы одна проверка не пройдена, запрос возвращает **403 Forbidden**.
